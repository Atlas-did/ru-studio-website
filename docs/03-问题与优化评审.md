# 问题与优化评审

这一份不讲结构，直接讲这项目里最值得注意的隐藏问题、设计亮点、以及后续最应该优先优化的地方。

## 一、设计得好的地方

### 1. 容错设计比较务实

[src/lib/api.ts](../src/lib/api.ts) 提供了 API 失败时的静态回退，这对展示站非常重要。它不是那种“接口挂了就全白屏”的脆弱实现，而是允许站点在弱网络或后端不可用时继续工作。

### 2. 结构拆分清楚

页面、区块组件、hooks、lib、后台页面和后端路由的边界都比较自然，后续做功能扩展时不容易把代码搅在一起。

### 3. 品牌感统一

首页的转场、滚动、光标、进度条、视觉区块都围绕同一套风格展开，不是东拼西凑的 UI 拼盘。这一点在品牌官网里是加分项。

## 二、设计不够好的地方

### 1. 没有测试体系

当前仓库里没有可见的自动化测试链路，复杂动效、数据回退、后台 CRUD 都很容易靠人工回归，维护成本会越来越高。

### 2. 数据库迁移过于手写

[server/db.js](../server/db.js) 里通过 `ALTER TABLE` 和启动时修补来处理历史数据，这在早期能用，但会越来越难管理。建议后续引入更明确的迁移策略，至少把 schema 演进和种子数据逻辑拆开。

### 3. 视觉效果偏重

项目里动效、噪点、转场、滚动监听、鼠标交互很多。如果这些效果都走 React 状态更新，性能风险会很快放大。视觉驱动的项目尤其要警惕“看起来高级，实际帧率不稳”。

### 4. `HashRouter` 有妥协痕迹

使用 `HashRouter` 可以减少部署复杂度，但 URL 不够干净，也不利于 SEO。如果后续要更正式地做品牌官网或搜索曝光，这里需要重新评估。

### 5. 目录里有历史残留

仓库里有两份重构备份目录，这会让搜索、阅读和新成员理解都变得更乱。它们不应被当成当前主线实现。

## 三、隐藏问题

### 1. API 失败掩盖真实错误

前端的静态回退能保证可用性，但也会掩盖一些真实问题。开发或排障时，如果只看页面是否能展示，容易忽略后端接口其实已经异常。

### 2. 数据覆盖风险

`useSiteData` 会把 API 数据和初始数据合并，这个设计能防止空字段覆盖内容，但如果后端数据结构不稳定，还是要小心“部分字段回填”带来的隐性问题。

### 3. 持久化风险

SQLite 文件和上传图片都依赖本地磁盘。若部署环境没有持久卷，重启或重新部署会造成数据丢失。

### 4. 运行时依赖敏感

项目对 Node 22 的特性有依赖，同时还保留了 `better-sqlite3` 的回退路径，说明运行环境一旦偏离预期，兼容性就可能出问题。

## 四、优先级最高的优化建议

1. 补测试，至少覆盖数据层和关键页面的基础回归。
2. 把数据库迁移逻辑整理成更明确的版本化方案。
3. 给高频交互组件做性能梳理，减少不必要的 React 重渲染。
4. 清理历史备份目录，减少文档和搜索噪音。
5. 如果准备长期运营，评估持久化存储和 SEO 路线。

## 五、如果只做三件事

如果当前只能优先做三项，我建议：

1. 补测试。
2. 整理数据库迁移。
3. 给高频交互和滚动监听做性能优化。
