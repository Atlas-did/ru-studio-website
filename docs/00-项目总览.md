# 项目总览

这个项目是一个典型的“视觉驱动 + 轻量 CMS”全栈网站：前端负责品牌展示、交互动效和路由切换，后端负责内容管理、登录鉴权、图片上传和 SQLite 持久化。它的设计目标很明确，不是通用后台系统，而是一个强调视觉气质和内容可编辑性的品牌官网。

## 一、主线结构

根目录下可以把主线理解成四层：

```text
app/
├─ src/        前端 React 应用
├─ server/     Express + SQLite 后端
├─ public/     静态资源与上传文件
├─ data/       数据库文件
└─ docs/       讲解文档
```

其中，`src/` 和 `server/` 是真正的业务核心；`public/` 和 `data/` 则是运行时依赖；`docs/` 是维护说明。

## 二、项目定位

- 前端不是纯静态展示，而是包含首页动效、作品集、日志、合作联系、礼物页、卦象页和管理后台。
- 后端不是复杂分层的业务系统，而是一个围绕品牌内容管理展开的服务端，重点在 CRUD、认证、上传和数据库初始化。
- 数据层采用 SQLite，部署成本低，适合中小规模内容站点，但对迁移和持久化要求更高。

## 三、入口文件

- [src/main.tsx](../src/main.tsx) 是前端挂载入口，只负责加载全局样式并渲染根组件。
- [src/App.tsx](../src/App.tsx) 是前端路由中枢，统一组织公开页和后台页。
- [server/index.js](../server/index.js) 是后端入口，负责中间件、路由、静态文件托管和 SPA 回退。
- [server/db.js](../server/db.js) 是数据库初始化与种子数据入口。

## 四、这个项目做得好的地方

1. 结构清楚。前端页面、区块组件、hooks、lib、后台页面分层比较明显，读代码时不容易迷路。
2. 视觉表达统一。首页动效、自定义光标、转场、滚动进度条和分段视觉都有明确的品牌语气。
3. 容错意识强。前端 API 层有回退数据，后端也有数据库初始化和默认种子逻辑，空环境下还能运行。
4. 后台能力完整。不是只做展示，而是保留了内容管理、媒体管理、配置管理这些实际维护入口。

## 五、做得不够好的地方

1. 工程化配套弱。当前没有测试体系，缺少迁移工具，数据库变更主要靠手写 `ALTER TABLE` 和启动时修补。
2. 历史遗留目录较多。仓库中存在两份重构备份目录，容易让搜索和阅读混乱。
3. 视觉代码偏重。项目大量依赖动效、滚动和过渡，如果不控制好 React 重渲染和 DOM 更新，容易出性能问题。
4. 说明文档分散。原有文档很多信息是重复的，总体上还缺一套按主题组织的阅读索引。

## 六、阅读建议

如果你第一次读这个仓库，建议按这个顺序：先看总览，再看前端阅读导览，接着看后端阅读导览，最后看问题与优化评审。这样能先建立结构感，再看细节，不容易被页面特效带偏。
