# 后端阅读导览

后端的目标很明确：服务前端展示和后台内容编辑，不追求复杂架构，而追求够用、稳定、易部署。

## 一、后端入口

后端从 [server/index.js](../server/index.js) 启动，主要做四件事：

1. 挂中间件，如 `helmet`、`compression`、`cors`、`morgan`、`rate limit`。
2. 挂静态资源和上传目录。
3. 挂公开 API 和后台 API。
4. 做 SPA 回退，保证前端路由在刷新时仍然可用。

## 二、数据库入口

[server/db.js](../server/db.js) 是后端最关键的状态入口。它会：

- 创建 `data/` 目录和 SQLite 文件。
- 优先尝试 Node 内置 SQLite 驱动，失败后退回 `better-sqlite3`。
- 创建 site_config、concepts、collection_items、journal_posts、contacts、admins、about_sections 等表。
- 启动时执行少量迁移和种子数据填充。

这种写法的优点是部署简单，缺点是数据库演进主要靠手写补丁，后期会越来越依赖启动时修补逻辑。

## 三、路由边界

后端 API 分成两大类：

- [server/routes/public.js](../server/routes/public.js) 负责站点公开数据、联系表单等。
- [server/routes/admin.js](../server/routes/admin.js) 负责管理后台登录、CRUD、上传等。

配合 [server/middleware/auth.js](../server/middleware/auth.js)，后台接口形成了一个比较完整的最小 CMS 闭环。

## 四、当前后端的优点

1. 启动即初始化，没数据库时也能自动创建结构和基础数据。
2. 代码路径短，排查问题比较快。
3. 对部署环境要求不算高，适合单机或轻量平台。
4. 公开接口和后台接口边界比较清楚。

## 五、当前后端的风险

1. 迁移机制偏弱。新增字段和旧数据修复都依赖手写 SQL，长期维护风险较高。
2. 默认管理账号和安全策略需要仔细控制，尤其是生产环境变量。
3. SQLite 适合轻量场景，但如果访问量、并发写入或多实例部署上来，风险会明显增加。
4. 上传文件和数据库文件都依赖本地持久化，部署平台如果不提供卷挂载，数据会有丢失风险。

## 六、建议阅读顺序

推荐这样看：

1. [server/index.js](../server/index.js)
2. [server/db.js](../server/db.js)
3. [server/routes/public.js](../server/routes/public.js)
4. [server/routes/admin.js](../server/routes/admin.js)
5. [server/middleware/auth.js](../server/middleware/auth.js)
