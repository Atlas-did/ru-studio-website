# 前端阅读导览

前端是这个项目最有辨识度的部分。它不只是把数据展示出来，而是通过路由、过渡动画、滚动交互和视觉组件，构建出一个很明确的品牌体验。

## 一、前端主入口

前端从 [src/main.tsx](../src/main.tsx) 启动，挂载到 `#root`，再进入 [src/App.tsx](../src/App.tsx)。`App.tsx` 使用 `HashRouter` 统一管理路由，并把公开页包在 `Layout` 和 `PageTransition` 中。

你可以把它理解成三层：

```text
main.tsx -> App.tsx -> Layout / PageTransition -> 具体页面
```

## 二、路由结构

公开页路由主要包括：

- `/` 首页
- `/about` 关于页
- `/collection` 作品收藏页
- `/collection/:slug` 作品详情页
- `/journal` 宣发日志页
- `/journal/:slug` 日志详情页
- `/cooperation` 合作联系页
- `/gift` 礼物页
- `/fortune` 卦象页

后台路由主要包括：

- `/admin/login`
- `/admin`
- `/admin/config`
- `/admin/about`
- `/admin/concepts`
- `/admin/collection`
- `/admin/journal`
- `/admin/media`

## 三、数据流

前端的数据层不是直接在页面里 fetch，而是通过 [src/lib/api.ts](../src/lib/api.ts) 统一封装。这个文件的关键点是“先尝试 API，失败就回退到静态数据”。

这一层的意义很大：

1. 本地开发时即使后端没起来，页面也能展示。
2. 部署或网络出问题时，网站不会直接空白。
3. 品牌展示类项目在演示场景下更稳。

`useSiteData` 负责把 API 返回的数据和初始数据合并，避免接口里某些字段为空时把原始内容覆盖掉。这个设计是对“后台数据不完整”的一个现实兜底。

## 四、组件组织

前端结构大致分成几类：

- `components/`：布局、转场、滚动进度、自定义光标、通用 UI。
- `sections/`：首页分区组件，适合承载较大的品牌展示区块。
- `pages/`：完整页面，既有公开页，也有管理后台页。
- `hooks/`：Lenis、滚动揭示、文本动画、站点数据等行为逻辑。
- `lib/`：API 封装、静态数据和通用工具。

这种拆法是合理的，原因是页面本身更偏内容编排，而动效和数据逻辑被抽出来后，维护成本会低很多。

## 五、设计亮点

1. 路由和布局拆得比较清楚，公开内容和后台内容没有混在一起。
2. 首页采用大区块拆分，适合视觉表达和局部调试。
3. `api.ts` + `useSiteData` 的组合让前端具备很强的离线容错能力。
4. 自定义交互组件的存在说明项目在“品牌感”上投入了足够注意力，不只是模板式页面拼装。

## 六、前端不足

1. 动画和滚动逻辑较多，天然更容易出现性能问题，尤其是 mousemove、scroll 这种高频事件。
2. 目前缺少测试，很难稳定验证动效回归、路由跳转和内容合并逻辑。
3. `HashRouter` 简化了部署，但 URL 观感和 SEO 都不够理想。
4. 某些视觉效果叠加过多时，容易出现“好看但重”的问题，后续要特别注意帧率和首屏体积。

## 七、建议阅读顺序

如果你要按代码理解前端，建议顺着这个顺序看：
